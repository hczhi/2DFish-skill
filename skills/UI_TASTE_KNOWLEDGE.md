# UI 审美知识库 · Design Taste Knowledge Base

这份文档是 /ui-review 功能的核心知识库。它沉淀了一位资深前端工程师的审美判断标准，用于指导 AI 对网页进行评分和输出修改建议。

维护方式：按维度分章节，每个维度写清楚"关注什么 → 什么算好 → 什么算烂 → 怎么改"。随时可以往任何章节追加新条目。

---

## 1. 排版层级 (Typography Hierarchy)

### 关注什么
字体选择、大小层级、字重对比、字间距、行高、最大行宽、标题与正文的视觉区分度。

### 什么算好
- 标题使用收紧字距（tracking-tighter），体现精致感和张力
- 字号层级清晰：H1 → H2 → H3 → body 有明确跳跃，不是线性递减
- 行高有节奏感：标题紧凑（leading-tight/none），正文舒适（leading-relaxed）
- 正文限制最大行宽（max-width 65ch），避免阅读疲劳
- 字体选择有品位：Geist、Satoshi、Inter（仅限功能性 UI）、Source Han Sans
- 字重对比明确：标题 600-700，正文 400，不要全页同一字重

### 什么算烂
- 标题巨大但没有收紧字距，看起来松散廉价
- 全页只用一种字号或字重，没有层级感
- 正文行宽超过 80ch，一行太长读不下去
- 随意混用三种以上字体家族
- 衬线体出现在 SaaS/工具类产品上（除非是编辑器/内容类产品）
- 中文字体使用系统默认宋体或微软雅黑，没有做任何优化

### 怎么改
- **层级重建**：建立 4 级字号体系（display: 3xl-6xl / heading: xl-2xl / body: base / caption: sm）
- **字距收紧**：所有 2xl 以上标题加 letter-spacing: -0.02em 到 -0.04em
- **行高配对**：标题 line-height: 1.1，正文 line-height: 1.6-1.7
- **行宽限制**：正文段落 max-width: 65ch
- **字体升级**：中文用思源黑体/鸿蒙字体，英文用 Geist/Inter/Outfit

## 2. 色彩和谐 (Color Harmony)

### 关注什么
色板收敛度、饱和度控制、背景与前景对比度、强调色使用节制、阴影颜色、冷暖一致性。

### 什么算好
- 全站最多 1 个强调色（accent），其余为中性色阶
- 饱和度统一且克制（< 80%），不刺眼
- 背景不用纯白 #fff，而是带微妙暖调（#fafafa、#f8f9fa）或冷调（#f1f5f9）
- 前景文字不用纯黑 #000，而是深灰（#1a1a1a、#18181b、zinc-900）
- 阴影颜色带环境色调，不是纯黑透明度
- 色彩系统有明确的 token：primary、secondary、muted、accent

### 什么算烂
- 多个高饱和色同时出现，视觉冲突
- 紫蓝霓虹渐变（典型的"AI 生成感"）
- 纯黑阴影（box-shadow 使用 rgba(0,0,0,x)）
- 冷灰和暖灰混用（如 gray-100 背景配 slate-800 文字）
- 强调色过度使用（到处都是品牌色，失去强调效果）
- 渐变文字用在大标题上（过度装饰，廉价感）
- 背景使用大面积高饱和渐变

### 怎么改
- **色板收敛**：选定 1 个中性色阶（zinc/slate/gray）+ 1 个强调色，全站锁定
- **饱和度降级**：将高饱和色的 saturation 降低 20-30%，融入整体色调
- **阴影着色**：box-shadow 的颜色从 rgba(0,0,0,0.1) 改为带色调的版本如 rgba(15,23,42,0.08)
- **对比度修复**：确保文字与背景对比度 ≥ 4.5:1（WCAG AA）
- **渐变克制**：如果要用渐变，限制在小面积装饰元素上，角度统一

## 3. 间距与留白 (Spacing & Whitespace)

### 关注什么
间距节奏感、padding 一致性、组件间距、section 间距、grid 对齐、呼吸感。

### 什么算好
- 使用 4px/8px 基准的间距系统（4、8、12、16、24、32、48、64、96）
- section 之间有充分的呼吸空间（py-16 到 py-24）
- 同级元素间距一致，不同层级间距有区分
- 卡片内部 padding 宽裕（p-6 到 p-8），不局促
- 留白是有意义的——用空间暗示分组关系
- 内容区与视口边缘有合理的 padding（px-4 到 px-8）

### 什么算烂
- 间距数值随意（13px、17px、22px），没有系统
- section 之间间距太小，内容拥挤在一起
- 卡片内部 padding 过小（p-2、p-3），文字贴边
- 元素之间间距不一致：同一行卡片的 gap 各不相同
- 大量内容堆积没有留白区域做视觉休息
- 移动端没有缩小间距，桌面端巨大的 padding 直接搬到手机上

### 怎么改
- **建立间距 scale**：全站统一使用 4/8/12/16/24/32/48/64/96 的倍数
- **section 节奏**：相邻 section 之间 py-16（移动端）到 py-24（桌面端）
- **内容分组**：相关元素间距小（gap-2 到 gap-4），无关元素间距大（gap-8 到 gap-12）
- **卡片呼吸**：内部 padding 至少 p-5，推荐 p-6 到 p-8
- **响应式缩放**：移动端间距为桌面端的 60-75%

## 4. 布局结构 (Layout Structure)

### 关注什么
布局多样性、视觉重心、对称/非对称选择、网格系统、内容层级是否通过布局体现。

### 什么算好
- 有明确的视觉重心和阅读动线
- 布局有变化：不是所有 section 都用同一种结构
- 非对称布局制造张力（6:4 分割、左文右图、错落排列）
- 使用 CSS Grid 构建可靠的网格系统
- 不同重要度的内容占据不同面积（重要的给更多空间）
- Hero section 有冲击力：大标题 + 清晰的 CTA，一屏内完成叙事

### 什么算烂
- 全页居中对称（text-center + items-center + 三等分卡片）—— 最典型的模板感
- 所有 section 都是"标题居中 + 下面三列卡片"的重复
- Hero 区域要滚很久才能看到核心内容
- 没有网格系统，元素用百分比和 calc 硬算位置
- 移动端只是把三列直接堆成三行，没有重新设计阅读顺序
- 内容层级扁平：所有卡片一样大、一样重要，没有主次

### 怎么改
- **打破居中**：Hero 改为左文右图 6:4 分割，标题左对齐
- **网格多样化**：特性展示区用 2 列 Z 字型或 bento grid，不要三等分
- **节奏变化**：交替使用"左图右文""全宽大图""两列对比"等不同 section 布局
- **层级体现**：重点内容用大卡片或全宽展示，次要内容用小卡片或列表
- **Grid 优于 Flex**：多列布局使用 grid-template-columns，避免 flex 百分比计算
- **一屏原则**：Hero 核心信息（标题 + 副标题 + CTA）必须在首屏可见

## 5. 视觉一致性 (Visual Consistency)

### 关注什么
圆角统一性、阴影风格一致性、图标风格一致性、按钮风格统一性、设计 token 的贯穿。

### 什么算好
- 全站圆角只有 2-3 个档位（如：4px 小元素、8px 卡片、12px 大容器）
- 阴影风格统一：同层级元素使用相同的阴影参数
- 图标风格一致：线条粗细（strokeWidth）全站统一
- 按钮系列完整：primary、secondary、ghost 三种状态，样式成体系
- 颜色使用 token 而非硬编码值，保证一致性
- 同类组件的视觉处理方式一致（如所有卡片的 hover 效果统一）

### 什么算烂
- 圆角混乱：rounded-sm 和 rounded-2xl 出现在相邻组件上
- 阴影风格不统一：有的组件用 shadow-sm，有的用 shadow-xl，还有的自定义
- 图标来自不同图标库，粗细风格各异
- 按钮样式随意：圆角按钮旁边出现直角按钮
- 间距在类似场景下不一致
- 有的卡片有边框，有的用阴影，有的什么都没有——同层级元素处理方式不同

### 怎么改
- **圆角锁定**：选定一套圆角体系（直角 / 软圆 8-12px / 全圆 9999px）并全站贯彻
- **阴影分级**：定义 3 级阴影（sm/md/lg），对应不同层级的组件
- **图标规范**：统一使用一套图标库，锁定 strokeWidth（推荐 1.5 或 2）
- **组件化思维**：相同功能的组件必须长得一样，不要每次重新设计
- **设计 token**：将颜色、圆角、阴影、间距定义为变量/token，避免硬编码

## 6. 整体质感与细节 (Aesthetics & Craft)

### 关注什么
是否有"精心打磨"的感觉、微交互质量、hover 状态完整度、过渡动画品质、背景处理、整体是否有记忆点。

### 什么算好
- 所有可交互元素都有 hover/active 状态（不是只有颜色变化）
- 过渡动画使用 ease-out 或 cubic-bezier，不是线性
- 按钮有触感反馈（:active 时微缩 scale-[0.98] 或 translateY(1px)）
- 背景不是纯色块，有微妙的纹理或网格增加质感
- 页面有"呼吸感"——不是塞满就好，留白本身是设计
- 加载状态使用骨架屏而非通用 spinner
- 分割线/边框使用低对比度（border-gray-100/200），不喧宾夺主

### 什么算烂
- hover 状态只是改了个颜色或加了 underline
- 没有任何过渡动画，状态切换是"跳变"的
- 按钮点击没有任何物理反馈
- 背景大面积纯色，感觉像未完成的 wireframe
- 使用默认的 loading spinner（圆圈转圈）
- 滚动时没有任何入场动画，所有内容一次性静态展示
- 表单输入框没有 focus 状态样式

### 怎么改
- **Hover 丰富化**：卡片 hover 加 translateY(-2px) + 阴影加深；按钮 hover 加背景色变化 + 微放大
- **过渡曲线**：统一使用 transition: all 0.2s cubic-bezier(0.16, 1, 0.3, 1) 
- **触感反馈**：按钮 :active 加 transform: scale(0.98) 或 translateY(1px)
- **背景质感**：加微妙的点阵网格（background-size: 40px 40px）或噪点纹理
- **入场动画**：使用 Intersection Observer 做滚动入场（fadeIn + translateY）
- **骨架屏**：加载状态匹配实际布局尺寸，使用微光动画
- **Focus 状态**：输入框 focus 时加 ring（ring-2 ring-primary/20）+ 微放大

## 7. 反模式清单 (Anti-Patterns)

这些是最常见的"一眼看出不专业"的设计问题：

### 布局反模式
| 模式 | 问题 | 修法 |
|------|------|------|
| 居中一切 | text-center + items-center 全页使用，模板感极强 | 左对齐为主，仅 CTA 区域允许居中 |
| 三等分卡片 | 特性展示永远是三列等宽卡片 | 换成 bento grid、Z 字型、或非对称双列 |
| 巨大空 Hero | Hero 区域只有一行标题 + 大量空白 | 加入产品截图、插图、或数据指标 |
| 所有 section 同构 | 每个区块都是"小标题 + 描述 + 网格" | 交替使用不同布局模式 |

### 色彩反模式
| 模式 | 问题 | 修法 |
|------|------|------|
| AI 紫蓝渐变 | from-purple-500 to-blue-500，典型 AI 审美 | 使用中性底色 + 单一克制强调色 |
| 纯黑阴影 | rgba(0,0,0,x) 做阴影，生硬无质感 | 阴影带入环境色调 |
| 过饱和强调色 | 品牌色饱和度 100%，刺眼 | 降低至 70-80%，与中性色融合 |
| 多种渐变混用 | 页面内出现 3+ 种渐变方向和色彩 | 最多一种渐变，且限制使用面积 |

### 排版反模式
| 模式 | 问题 | 修法 |
|------|------|------|
| 字号无层级 | 标题和正文大小差异不明显 | 建立 4 级字号体系，跳跃明确 |
| 字距未收紧 | 大标题字间距宽松，显得松散 | 2xl 以上加 tracking-tighter |
| 行宽过长 | 一行正文超过 80 个字符 | max-width: 65ch 限制 |
| 滥用大写 | 全大写标题 + tracking-widest 到处都是 | 最多每 3 个 section 用一次眉题 |

### 质感反模式
| 模式 | 问题 | 修法 |
|------|------|------|
| 无 hover 状态 | 可交互元素没有视觉反馈 | 补全 hover/active/focus |
| 跳变过渡 | 状态切换没有动画 | 加 transition 200-300ms |
| 圆角混乱 | 同一页面 4+ 种圆角值 | 锁定 2-3 个档位 |
| 默认 spinner | 用原生圆圈转圈做加载 | 改为匹配布局的骨架屏 |

## 8. 行业基准 (Industry Benchmarks)

不同类型产品的设计标准不同：

### SaaS / 工具类
- 标杆：Linear、Vercel、Notion、Raycast
- 特征：极度克制、功能优先、信息密度适中、无装饰噪点
- 字体：无衬线（Geist、Inter、SF Pro）
- 色彩：中性色底 + 单一强调色
- 圆角：统一 8-12px
- 布局：功能区块清晰，不追求花哨

### 电商 / 消费品
- 标杆：Apple Store、Shopify、Glossier
- 特征：产品图片为核心、留白慷慨、排版有呼吸感
- 字体：可适度使用衬线体做品牌感
- 色彩：品牌色突出但不过度
- 布局：大图 + 简洁文案，突出商品本身

### 官网 / 品牌站
- 标杆：Stripe、Linear Marketing、Arc Browser
- 特征：视觉冲击力强、动效丰富、叙事性布局
- 字体：标题可大胆（display font），正文克制
- 色彩：允许更大胆的强调色和渐变（但要克制）
- 布局：非对称、交错、视差滚动

### 内容 / 博客类
- 标杆：Medium、Substack、个人技术博客
- 特征：阅读体验优先、排版极致、最小化干扰
- 字体：正文可用衬线体提升阅读感
- 色彩：极简，接近黑白灰
- 布局：单栏为主，行宽严格控制

## 9. 修改建议的分级体系

生成修改建议时，按严重程度和改动范围分三级：

### Level 1: CSS 调优
只改样式属性值，不动 HTML 结构。
- 修改间距、字号、颜色、阴影、圆角、字重
- 添加 hover/transition/animation
- 调整响应式断点下的样式

示例：
```
[CSS调优] 主标题
选择器：.hero h1
现状：font-size: 48px, letter-spacing: normal, font-weight: 700
改为：font-size: 48px, letter-spacing: -0.03em, font-weight: 600
原因：字距收紧让大标题更有张力和精致感，字重从 700 降到 600 避免过于厚重
```

### Level 2: 结构调整
需要调整 HTML 结构，但不改变页面整体布局逻辑。
- 调整元素嵌套层级
- 添加 wrapper 容器优化间距
- 重排元素顺序
- 拆分过大的组件

示例：
```
[结构调整] 特性展示区
选择器：.features-grid
现状：三个 feature-card 并排，每个内部是 icon + title + description 垂直排列
改为：每个 card 改为 icon 左对齐 + 右侧 title/description 的水平布局，增加辨识度
原因：纯垂直排列的三列卡片是最典型的模板感布局，水平布局增加阅读效率和视觉变化
代码模式：display: grid; grid-template-columns: 48px 1fr; gap: 16px; align-items: start;
```

### Level 3: 布局重构
需要重新设计 section 的整体布局逻辑。
- 改变栏数、分割比例
- 重新设计 section 的叙事结构
- 改变视觉重心和阅读动线
- 替换整个布局模式

示例：
```
[布局重构] Hero Section
现状：居中标题 + 居中副标题 + 居中按钮 + 下方三列特性卡片。典型的模板首页，毫无记忆点。
改为：左右 6:4 非对称分割。左侧：标题左对齐（tracking-tighter）+ 副标题（max-w-[480px]）+ CTA 按钮组。右侧：产品截图或 3D 插图，微微上浮（translateY）。
原因：非对称布局制造视觉张力，左对齐标题更有力量感，打破"居中一切"的 AI 默认审美
参考：Linear 首页、Vercel 首页的非对称 Hero 布局
```

## 10. AI 执行指令的输出规范

输出给用户复制到 Cursor/Claude 的修改指令，必须满足：

### 格式要求
1. 每条指令独立、原子化，一条只改一个问题
2. 明确标注修改级别：[CSS调优] / [结构调整] / [布局重构]
3. 必须包含：选择器/目标区域 + 现状描述 + 目标状态 + 原因
4. 如果涉及代码改动，给出具体的 CSS 或 HTML 片段
5. 按优先级排序：影响最大的排在最前面

### 语言风格
- 用"指令"语气，不用"建议"语气——这是给 AI 执行的任务清单
- 具体到选择器和属性值，不说"优化间距"而说"padding: 16px → 32px"
- 说明"为什么改"而不只是"改什么"——让执行的 AI 理解设计意图
- 如果是布局重构，描述目标布局的视觉效果，让 AI 知道做对了是什么样

### 开头模板
```
你是一个高级前端工程师，需要按以下清单优化页面的视觉设计。
每条改动都有明确的选择器和目标值，请逐条执行。
不要改动业务逻辑和数据绑定，只调整视觉呈现。

页面技术栈：[Vue/React/HTML] + [Tailwind/CSS]
当前页面核心问题：[1-2句概括]

---

[修改清单从这里开始]
```

## 维护指南

### 如何添加新内容
1. 找到对应的维度章节
2. 在"什么算烂"下面加一条你发现的新问题
3. 在"怎么改"下面加对应的修复方法
4. 如果是全新的模式，加到"反模式清单"表格里

### 什么值得记录
- 你在实际项目中踩过的坑
- 你看到烂页面时的第一反应（那个直觉就是判断标准）
- 好设计中你注意到的细节（那些让你觉得"精致"的原因）
- 客户/用户对优化结果的反馈（特别是"这里变好了"和"这里还是不对"）

### 什么不需要记录
